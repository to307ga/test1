# EC2要件定義書

## 1. プロジェクト概要

### 1.1 前提
- **goo/OCNサービスでの標準的なシステム構成**: Webアプリケーション環境のEC2インスタンスベース構成を自動構築
- **セキュリティ**: WAF + Security Groupsによる多層防御
- **コンピューティング**: Amazon EC2によるWebアプリケーション実行
- **Infrastructure as Code**: CloudFormation + Sceptre + Ansible + Jenkinsによる自動化

### 1.2 プロジェクト名
AWS標準構成テンプレート（EC2版）

### 1.3 目的
AWS上にセキュアでスケーラブルなWebアプリケーション環境を、EC2インスタンスをベースとして構築するための標準構成テンプレートの提供

### 1.4 対象環境
- **環境名**: 本番、テスト、開発環境
- **リージョン**: ap-northeast-1（東京）
- **コンピューティング**: Amazon EC2（オンプレミスサーバからの移行を想定）

### 1.5 EC2採用の背景
- **リフト&シフト移行**: オンプレミス環境からのスムーズな移行
- **既存アプリケーション継続利用**: 既存のWebアプリケーションをそのまま稼働
- **運用ノウハウの活用**: 既存のサーバ運用体制・ノウハウの継続活用
- **段階的モダナイゼーション**: バッチサーバのLambda移行、DBサーバのAurora MySQL移行を実施
- **Immutableインフラ**: ゴールデンAMIによるImmutableインフラストラクチャの実現

### 1.6 各サービス担当の主な検討項目
- **環境構成**: 検証/開発環境の有無の検討
- **アカウント構成**: 検証/開発アカウントの有無の検討
- **システム構成**: WG提供テンプレートに含まれないリソースの設計
- **運用体制**: OSパッチ管理、セキュリティ対応の運用体制の確立

## 2. システム構成要件

### 2.1 AWSアカウント構成要件
- **本番環境**: 独立したアカウントを必須とする
- **検証/開発環境**: 各サービス担当で検討（独立アカウント or 同一アカウント）
- **アカウント払い出し**: CCoEにて払い出し
- **環境分離**: システムセキュリティ対策チェックリスト8.2に基づく分離

### 2.2 Webアプリケーション構成要件
- **基盤**: Amazon EC2によるWebアプリケーション実行環境
- **基本構成要素**: CloudFront、ELB、EC2、S3、Aurora MySQL
- **OS**: Amazon Linux 2023（決定）
- **スケーリング**: 手動スケーリング対応（必要時に手動でインスタンス追加）
- **ロードバランシング**: Application Load Balancer（ALB）による負荷分散
- **セッション管理**: スティッキーセッション対応
- **Launch Template**: EC2インスタンスの標準化された起動設定を定義するテンプレート
  - AMI ID、インスタンスタイプ、セキュリティグループ、IAMロール等の起動パラメータを一元管理
  - バージョン管理により設定変更の履歴を保持
  - Auto ScalingやBlue/Greenデプロイメントでの起動設定として利用

### 2.3 EC2インスタンス要件

#### 2.3.1 インスタンスタイプ
- **採用インスタンス**: 
  - **開発環境**: t3.medium（vCPU 2コア、メモリ 4GB）
  - **本番環境**: t3.2xlarge（vCPU 8コア、メモリ 32GB）
- **推奨ファミリー**: 汎用インスタンス（T3/T4g、M5/M6i）
- **最小スペック**: vCPU 2コア、メモリ 4GB（開発環境）
- **選定基準**: オンプレミス環境のスペックとワークロード特性を考慮
- **購入オプション**: 
  - Immutableインフラ（頻繁な作成・削除）を考慮
  - Blue/Greenデプロイ時の停止インスタンス増加を考慮
  - EC2 Instance Savings Plans（特定インスタンスファミリーに対する割引、最大72%）の活用を検討
  - インスタンス数の増加は基本的に予定されていないため、柔軟性よりも割引率重視

#### 2.3.2 ストレージ要件
- **ルートボリューム**: EBS gp3、最小30GB、暗号化必須
- **データボリューム**: 用途に応じてgp3またはio2を選択、暗号化必須
- **スナップショット**: 定期的な自動バックアップ（日次、7日間保持）
- **ボリューム最適化**: 使用状況に応じた容量・IOPS調整

#### 2.3.3 ネットワーク要件
- **配置**: プライベートサブネットへの配置必須
- **ENI**: 各インスタンスに専用のElastic Network Interface
- **セキュリティグループ**: 最小権限の原則に基づく通信制御
- **Elastic IP**: 必要に応じて固定IPアドレスの割り当て（NATゲートウェイ経由推奨）

### 2.4 CDN構成要件
- **CDNサービス**: CloudFrontを前提とする
- **構成要素**: WAF、CloudFront、S3等を含む
- **静的コンテンツ**: S3バケットからの配信
- **SSL/TLS**: 自動証明書管理（ACM）

### 2.5 データベース機能要件
- **データベース**: Amazon Aurora MySQL
- **バージョン戦略**: 
  - Aurora MySQL 3.x系（MySQL 8.0互換）の最新LTS（Long-Term Support）版を採用
  - 現在の推奨: Aurora MySQL 3.10.x（LTS release、最低3年間サポート）
  - LTSバージョンは新機能追加なし、重要な修正のみを含むパッチレベルで提供
  - 年次でのパッチレベル更新を推奨（セキュリティ・運用修正の適用）
  - 自動マイナーバージョンアップグレードは無効化（LTS維持のため）
- **高可用性**: マルチAZ配置（3つのAZ）
- **バックアップ**: 7日間の自動バックアップ
- **暗号化**: 保存時暗号化対応（マネージドキー）
- **接続管理**: Secrets Managerによる認証情報管理
- **スケーリング**: スペックアップ、スケールアウト対応



## 3. セキュリティ要件

### 3.1 アクセス制御要件
- **IAMユーザ管理**: 手動パスワード払い出し、パスワードポリシー設定
- **ロールベースアクセス**: 管理者用ロール、参照用ロール等の権限別ロール
- **MFA必須**: 初回ログイン時のMFA設定必須
- **ネットワーク制御**: NetworkFirewall、SecurityGroup、WAFのIPセットによる多層防御
- **EC2アクセス**: AWS Systems Manager Session Managerによる安全なアクセス（SSH踏み台不要）
  - **基本方式（推奨）**: IAMロールベースのアクセス制御
    - Session Managerはデフォルトで`ssm-user`アカウントを使用
    - IAMポリシーでSession Manager操作権限を制御（最小権限の原則）
    - CloudTrailでIAMユーザー/ロール単位のアクセス記録
    - シンプルで管理しやすく、AWSのベストプラクティスに準拠
  - **OSユーザーアカウント管理（オプション）**:
    - 監査要件でOS層のユーザー識別が必要な場合検討
    - rbash対応が必要な場合検討
    - Session Managerの「Run As」機能によるIAMとOSユーザーの紐付け
    - IAMエンティティ（ユーザー/ロール）に`SSMSessionRunAs`タグを設定
    - OSレベルの監査ログでユーザー操作を追跡可能
    - rootユーザーでの接続は不可
    - **ec2-userセキュリティ対策（以下のいずれかを選択）**:
      - **案1（推奨）**: アカウント無効化
        - `sudo usermod -L ec2-user && sudo usermod -s /usr/sbin/nologin ec2-user`
        - 完全にログイン不可、最も安全
      - **案2**: ログイン時アラート監視
        - ec2-userログインを許可するが、ログイン検知時に即座にアラート発火
        - CloudWatch Logs Insightsでec2-userログインを監視
        - SNS通知でセキュリティチームに即座に通知
        - 監査ログで事後追跡可能
        - 緊急時アクセス用として残すが、厳格な監視下に置く

### 3.2 通信制御要件
- **通信経路制御**: 一般ユーザ → CloudFront → ELB → EC2 → Aurora,ElastiCache
- **WAF設定**: CloudFront用WAF、ELB用WAFのIPセットによるアクセス制限
- **SecurityGroup**: 各層での適切な通信制限
- **カスタムヘッダ**: ELBのカスタムヘッダによるCloudFrontからのアクセス許可
- **アウトバウンド制御**: 必要最小限の外部通信のみ許可

### 3.3 暗号化要件
- **通信暗号化**: HTTPS通信（一般ユーザ → CloudFront、CloudFront → ELB）
- **証明書管理**: ACMによるSSL証明書の発行・自動更新
- **データ暗号化**: 各サービスでのマネージドキーによるサーバーサイド暗号化
  - EC2: EBSボリュームの暗号化
  - Aurora: 保存時暗号化
  - ElastiCache: 保管時・転送時暗号化
  - S3: 保存時暗号化
- **転送時暗号化**: EC2-Aurora間、EC2-ElastiCache間の通信暗号化

### 3.4 セキュリティ監視要件
- **EDR**: CrowdStrikeによるエンドポイントセキュリティ監視
- **脆弱性対策**: CloudFront前段のWAF設定
- **脆弱性検知**: Systems Manager Patch Manager、Inspector、feature vulsによる脆弱性検知
- **DDoS対策**: AWS Shield Advanced、WAFレートベースルール、Bot Control
- **侵入検知**: VPCフローログ、CloudWatch Logsによる異常検知

### 3.5 監査証跡要件
- **アクセスログ**: ALBアクセスログ、VPCフローログ、WAFログの取得
- **API呼び出しログ**: CloudTrailによるAPI呼び出しログの取得
- **OSログ**: CloudWatch Logs AgentによるOS/アプリケーションログの収集
- **ログ保管**: システムセキュリティ対策チェックリストに基づく保管期間

### 3.6 OSセキュリティ要件
- **パッチ管理**: Systems Manager Patch Managerによる自動パッチ適用
- **パッチ適用タイミング**: 無停止で平日営業時間内に実施
- **緊急パッチ**: 重大な脆弱性発見時の即時適用手順の確立
- **アンチウイルス**: CrowdStrikeによるリアルタイム保護
- **ハードニング**: CIS Benchmarkに基づくOS設定の強化
  - **CIS Benchmark**: Center for Internet Security（CIS）が公開するセキュリティ設定基準
  - OSやミドルウェアのセキュアな設定のベストプラクティス集
  - 不要なサービスの無効化、ファイルパーミッションの適正化、監査ログ設定等を含む
  - Amazon Linux 2023向けCIS Benchmarkに準拠した設定を適用
- **ログ監視**: 不正アクセス、権限昇格の検知

## 4. 非機能要件

### 4.1 可用性要件
- **可用性目標**: 99.9%以上
- **マルチAZ構成**: 3つのAZ（1a, 1c, 1d）を使用した冗長化
- **手動フェイルオーバー**: 障害時の手動によるインスタンス追加・切り離し
- **ヘルスチェック**: ELBヘルスチェックによる異常インスタンスの検知と手動対応

### 4.2 拡張性要件
- **手動スケーリング**: 必要に応じて手動でEC2インスタンスを追加・削除
- **スケールアウト**: 手動でインスタンスを追加して台数増加
- **スケールイン**: 手動でインスタンスを削減して台数減少
- **Auroraスケーリング**: スペックアップ、スケールアウト対応
- **負荷分散**: ELBによる自動負荷分散
- **起動時間**: Launch Templateとゴールデンイメージによる高速起動（5分以内）

### 4.3 パフォーマンス要件
- **レスポンス時間**: ページ表示2秒以内
- **スループット**: 同時接続数1000以上
- **スケーラビリティ**: 負荷に応じた自動スケーリング
- **CDN活用**: CloudFrontによる静的コンテンツ配信最適化
- **インスタンス起動時間**: Launch Templateによる高速起動（5分以内）

### 4.4 運用要件

#### 4.4.1 監視要件
- **監視ツール**: Datadog、CloudWatch、Systems Managerによる包括的監視
- **メトリクス監視**: CPU、メモリ、ディスク、ネットワークの継続監視
- **ログ監視**: アプリケーションログ、アクセスログ、OSログの収集・分析
- **アラート**: 異常検知時の自動通知
- **ダッシュボード**: リアルタイム運用状況の可視化

#### 4.4.2 パッチ管理要件
- **定期パッチ適用**: Systems Manager Patch Managerによる月次パッチ適用
- **適用タイミング**: 無停止、平日営業時間内に実施（Blue/Greenデプロイ活用）
- **パッチベースライン**: セキュリティパッチの自動適用、その他パッチの検証後適用
- **ロールバック**: パッチ適用後の問題発生時のロールバック手順
- **パッチレポート**: 適用状況の定期レポート作成

#### 4.4.3 バックアップ要件
- **ゴールデンAMI**: パッチ適用・アプリデプロイ時に検証済みAMIを作成
  - AMIをゴールデンイメージとして使用（事前検証済み）
  - デプロイ前に必ずAMI作成（ロールバック用）
  - 不定期作成（パッチ公開、アプリリリースのタイミング）
- **世代管理**: 過去10世代のAMI保持（月次パッチ＋アプリリリースで2-3世代/月想定）
- **EBSスナップショット**: AMI作成時に自動的にEBSスナップショットも作成される
- **Auroraバックアップ**: 自動バックアップ（毎日 18:00-18:30）
- **バックアップ検証**: 定期的なリストアテスト実施

#### 4.4.4 デプロイ要件
- **デプロイ方式**: Blue/Greenデプロイメント、ローリングアップデート対応
- **自動化**: Launch TemplateとAnsibleによる自動セットアップ
  - Launch Templateで基本的な起動設定を定義
  - Ansibleでミドルウェア設定・アプリケーション配置を自動化
  - ユーザーデータは最小限（Ansibleエージェント起動等）
- **設定管理**: Systems Manager Parameter StoreまたはSecrets Managerによる設定管理
  - **Parameter Store**: 非機密な設定値の管理（接続先URL、ポート番号、設定フラグ等）
    - 暗号化オプション有り、KMS統合
    - バージョン管理、階層構造での管理
    - CloudFormation、Lambda等から参照可能
  - **Secrets Manager**: 機密情報の管理（DBパスワード、APIキー等）
    - 自動ローテーション機能
    - マルチリージョン複製
    - RDS/Aurora等との自動統合
- **アプリケーションデプロイ**: CodeDeployまたはAnsibleによる自動デプロイ

#### 4.4.5 ライフサイクル管理
- **Golden AMI**: 標準化されたベースAMIの作成・管理
  - パッチ適用・アプリ更新時に検証環境で検証済みAMIを作成
  - 検証済みAMIをゴールデンイメージとして本番デプロイに使用
- **AMI更新サイクル**: 月次でのセキュリティパッチ適用済みAMI更新
- **世代管理**: 過去10世代のAMI保持
- **古いリソース削除**: 未使用のAMI、スナップショットの定期削除（11世代目以降）

## 5. 技術要件

### 5.1 インフラストラクチャ
- **コンピューティング**: Amazon EC2、AWS Lambda（バッチ処理）
- **ネットワーク**: Amazon VPC
- **ロードバランサー**: Application Load Balancer
- **データベース**: Amazon Aurora MySQL
- **ストレージ**: Amazon EBS、Amazon S3
- **CDN**: Amazon CloudFront

### 5.2 管理・運用サービス
- **システム管理**: AWS Systems Manager（Session Manager、Patch Manager、Parameter Store）
- **設定管理**: 
  - **Launch Template**: EC2インスタンスの起動設定テンプレート（AMI、インスタンスタイプ、SG、IAMロール等）
  - **EC2 Image Builder**: ゴールデンAMIの自動作成・更新パイプライン（オプション）
    - パッチ適用やソフトウェアインストールを自動化
    - 定期的な自動ビルドとテスト実行
    - バージョン管理された一貫性のあるAMI作成
- **デプロイ**: AWS CodeDeploy、Ansible、Jenkins
- **自動化**: CloudWatch Events/EventBridge
- **バックアップ**: AMI、EBS Snapshots

### 5.3 セキュリティ
- **WAF**: AWS WAF
- **ファイアウォール**: AWS Network Firewall
- **証明書**: AWS Certificate Manager
- **認証情報管理**: AWS Secrets Manager
- **EDR**: CrowdStrike Falcon
- **脆弱性検知**: Amazon Inspector、Systems Manager、feature vuls
- **アクセス管理**: AWS Systems Manager Session Manager

### 5.4 監視・ログ
- **監視**: Datadog、Amazon CloudWatch
- **ログ収集**: CloudWatch Logs Agent、Fluent Bit
- **ログ分析**: Amazon Athena
- **アラート**: Amazon SNS
- **API監査**: Amazon CloudTrail

## 6. 制約事項

### 6.1 技術的制約
- **リージョン**: ap-northeast-1のみ
- **AZ**: 3つのAZ（1a, 1c, 1d）を使用
- **IPアドレス**: 10.60.0.0/22のCIDRブロック
- **OS**: Amazon Linux 2023またはRHEL系の最新安定版を推奨
- **CDN**: CloudFrontを前提とする

### 6.2 セキュリティ制約
- **MFA必須**: 初回ログイン時のMFA設定必須
- **HTTPS通信**: 外部向けシステムとしてHTTPS通信必須
- **マルチAZ**: 高可用性のためマルチAZ構成必須
- **暗号化**: 通信時・保存時暗号化必須
- **パッチ適用**: セキュリティパッチの定期適用必須（月次以上）
- **EDR導入**: エンドポイント保護の導入必須

### 6.3 運用制約
- **メンテナンス時間**: 無停止、平日営業時間内に実施（Blue/Greenデプロイ活用）
- **バックアップ時間**: 
  - AMI作成: 不定期（パッチ適用・アプリデプロイ時）
  - Auroraバックアップ: 毎日 18:00-18:30
- **証明書更新**: 自動更新（ACM）
- **ログ保管**: システムセキュリティ対策チェックリストに基づく期間
- **パッチ適用**: 月次定期実施、緊急時は臨時実施（平日営業時間内、無停止）

- 静的コンテンツが正常に配信されること
- キャッシュ機能が正常に動作すること

- セキュリティ要件を満たすこと
- パフォーマンス要件を満たすこと
- 監視・アラートが正常に動作すること
- マルチAZ構成による冗長化が実現されること
- パッチ適用の自動化が実現されること

### 7.3 運用面
- Systems Managerによる安全なアクセスが実現されること
- Patch Managerによる自動パッチ適用が動作すること

## 8. 前提条件

### 8.1 環境前提
- AWSアカウントが利用可能であること
- 必要なIAM権限が付与されていること
- ドメイン名が取得済みであること
- CCoEによるアカウント払い出しが完了していること

### 8.2 技術前提
- アプリケーションがEC2環境で動作可能であること
- アプリケーションコードがGitリポジトリに格納されていること
- SSL証明書が取得済みであること（ACMまたは外部証明書）
- Datadogアカウントが利用可能であること
- CrowdStrikeライセンスが利用可能であること

### 8.3 運用前提
- OSパッチ管理の運用体制が確立されていること
- セキュリティインシデント対応体制が確立されていること
- 24時間365日の監視体制が確立されていること（または検討されていること）
- バックアップからのリストア手順が確立されていること

### 8.4 セキュリティ前提
- システムセキュリティ対策チェックリストに準拠していること
- MFA設定が可能な環境であること
- ログ保管期間が規定に準拠していること
- パッチ適用の承認プロセスが確立されていること

## 9. 除外事項

### 9.1 機能除外
- 外部システムとの連携機能
- リアルタイム通信機能
- 大容量ファイルアップロード機能（EC2のストレージ制約を考慮）
- オンプレミス環境との連携機能

### 9.2 非機能除外
- 99.99%以上の可用性
- グローバル展開（マルチリージョン）
- オンプレミス環境とのハイブリッド構成
- 99.99%以上のSLA保証

### 9.3 運用除外
- 自動的なOSメジャーバージョンアップグレード
- アプリケーションレベルの自動最適化
- カスタムミドルウェアの自動管理
- 一部ミドルウェア（Apache、Java等）の自動メジャーバージョンアップグレード

## 10. EC2採用時の主な検討事項

### 10.1 運用負荷に関する検討事項
以下の運用業務が発生するため、運用体制の確立が必要です：
- **OSパッチ管理**: 定期的なセキュリティパッチ適用とリブート管理
- **ミドルウェア管理**: Webサーバー、アプリケーションサーバーの更新管理
- **キャパシティプランニング**: インスタンスタイプ・台数の定期見直し
- **AMI管理**: ゴールデンAMIの作成・更新・世代管理
- **インスタンス更新**: 新AMIへのインスタンス入れ替え作業

### 10.2 セキュリティに関する検討事項
EC2を採用する場合、以下のセキュリティ対策の実装・運用が必要です：
- **OSレベルの脆弱性対策**: Systems Manager、Inspectorによる継続的な脆弱性管理
- **EDR導入**: CrowdStrike等のエンドポイント保護ソリューションの導入
- **ログ監視**: CloudWatch Logs、Datadogによる異常検知
- **侵入検知**: 不正アクセス、権限昇格の監視
- **コンプライアンス**: CIS Benchmark等のセキュリティ基準への準拠

### 10.3 コストに関する検討事項
コスト最適化のため、以下を検討してください：
- **リザーブドインスタンス**: 1年または3年の予約による割引活用
- **Savings Plans**: Compute Savings Plansによる柔軟なコスト削減
- **インスタンスタイプ最適化**: ワークロードに最適なインスタンスタイプの選定
- **Auto Scaling設定**: 過剰なスケールアウトの防止
- **スポットインスタンス**: 開発/検証環境での活用検討

### 10.4 ECS/Fargateとの比較検討
EC2採用前に、以下の観点でECS/Fargateとの比較を推奨します：

| 観点 | EC2 | ECS/Fargate |
|------|-----|-------------|
| 運用負荷 | OSパッチ管理が必要 | OSレス、パッチ管理不要 |
| スケーラビリティ | Auto Scalingによる柔軟な対応 | コンテナ単位の高速スケール |
| セキュリティ | OS層の管理が必要 | コンテナ層の管理のみ |
| コスト | 長期利用で割安（RI/SP活用） | 短期・変動負荷で効率的 |
| 学習コスト | 従来型、ノウハウ豊富 | コンテナ技術の習得必要 |
| デプロイ速度 | インスタンス起動に時間 | コンテナ起動が高速 |

## 11. システム設計時の検討項目（ベストプラクティス考慮）

本要件を満たすシステム設計を行う際、以下の項目について詳細設計書で具体的な設計を行ってください。これらはAWS Well-Architected FrameworkのEC2ベストプラクティスを考慮した検討事項です。

### 11.1 設計書での検討が必要な項目
- **インスタンスタイプ最適化**: ワークロード特性に基づく最適なインスタンスファミリー選定
- **AMI管理戦略**: ゴールデンAMIのパイプライン構築（EC2 Image Builder活用）
- **パッチ管理自動化**: Systems Manager Patch Managerの詳細設定
- **Auto Scaling最適化**: スケーリングポリシー、ターゲット追跡、ヘルスチェック設計
- **起動最適化**: ユーザーデータスクリプト、CloudFormation Init、Launch Template設計
- **セキュリティ強化**: IMDSv2強制、VPCエンドポイント活用、セキュリティグループ最小化
- **監視・ロギング**: 統合監視基盤設計、ログAggregation設計
- **ディザスタリカバリ**: バックアップ・リストア手順、クロスリージョンバックアップ検討
- **コスト最適化**: リザーブドインスタンス戦略、タグ付け戦略

### 11.2 運用設計書での検討が必要な項目
- **運用自動化**: Systems Managerによる運用タスク自動化
- **パッチ適用手順**: 定期メンテナンス、緊急パッチ適用フロー
- **インシデント対応**: 障害検知、エスカレーション、復旧手順
- **キャパシティ管理**: リソース使用状況レビュー、キャパシティプランニングサイクル
- **変更管理**: AMI更新、設定変更の承認プロセス
